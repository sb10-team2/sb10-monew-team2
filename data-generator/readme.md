# 🚀 Data Generator 모듈 개요 (부하 테스트용)

## 1. 모듈 분리 배경 및 이유 (Why?)

기존에는 데이터 생성 로직이 `test` 패키지 내부에 구현되어 있었으나, 다음과 같은 아키텍처적 고민을 통해 별도의 모듈로 분리하게 되었다.

* **목적의 불일치:** '대용량 더미 데이터 생성'이라는 행위가 순수 애플리케이션 '테스트(Test)'의 범주에 속하는가에 대한 의미론적 의문 제기.
* **불필요한 오버헤드 제거:** 대규모 데이터를 DB에 적재하는 작업에는 톰캣(Tomcat) 웹 서버나 JPA(영속성 컨텍스트) 같은 무거운 프레임워크 기능이 필요하지 않음.
* **최적화:** 데이터 생성에 특화된 가벼운 환경(예: JdbcTemplate 활용 등)을 구축하여 빠르고 효율적으로 동작하게 만들기 위해 독립 모듈로 분리.

---

## 2. 패키지 구조의 변화 (Architecture)

기존 단일 프로젝트 구조에서, 핵심 비즈니스 로직 / 데이터 생성 / 부하 테스트(k6) 역할을 명확히 나누는 **멀티 모듈 구조**로 변경.

**[변경 후 프로젝트 구조]**  
```
root
├── src/ # 기존 프로젝트, 구조 변경 없음
├── data-generator/ # (신규 모듈) JPA, 톰캣 없이 빠르고 가볍게 더미 데이터만 생성 및 DB 적재
├── load-test/ # (신규 모듈) k6를 활용한 실제 부하 테스트 스크립트 및 시나리오 관리
└── monitoring/ # 시스템 메트릭 모니터링을 위한 모듈
```

## 3. 엔티티별 데이터 생성 타겟팅 (Data Scale)

피드백을 반영하여 시스템 부하 및 테스트 효율을 고려해 전체적인 생성 건수를 하향 조정했습니다. *(참고: 데이터 볼륨 및 매핑 비율에 일부 논리적 예외 상황을 허용함)*

| 엔티티 (Entity)        | 생성 규모 및 정책                                                                                           |
|:--------------------|:-----------------------------------------------------------------------------------------------------|
| **User**            | **1,000명** (기존 10,000명에서 축소)                                                                         |
| **Comment**         | **50,000개** (User 당 50개 작성)                                                                          |
| **CommentLike**     | **50,000개** (User 당 50개 좋아요)                                                                         |
| **Interest**        | **1,000개** (기존 10,000개에서 축소)                                                                         |
| **InterestKeyword** | **관심사 당 3개** (기존 10개에서 축소)                                                                           |
| **Subscription**    | **User 당 3개** (기존 10개에서 축소)                                                                          |
| **NewsArticle**     | **50,000개** (기존 1,000,000개에서 대폭 축소)                                                                  |
| **ArticleView**     | **User 당 10개** (기존 100개에서 축소)                                                                        |
| **ArticleInterest** | **평균 약 50,000개** (관심사 1개당 연관 기사를 1~100개 랜덤으로 매핑)                                                     |
| **Keyword**         | **1,000개** (기존 10,000개에서 축소)                                                                         |
| **Notification**    | **총 26,000개** <br> - 댓글 좋아요 알림: 1,000개 (1:1 매핑) <br> - 관심사 기반 알림: 25,000개 (User당 25개 구독 가정 × 1,000명) |
